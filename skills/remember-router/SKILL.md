# Remember Router — 기억해 자동 라우팅 시스템

## 의존성

| 의존성 | 종류 | 필수 여부 | 설명 |
|--------|------|----------|------|
| Obsidian Vault | 파일시스템 | 필수 (표준 모드) | `{OBSIDIAN_VAULT}` 경로 — L3-Active/, L1-Knowledge/, Action Tracker.md 필요 |
| Google Calendar MCP | MCP | 선택 | 일정/약속 저장 시 `gcal_create_event` 사용. 없으면 캘린더 저장 생략 |
| Claude Memory | 파일시스템 | 선택 | `~/.claude/projects/…/memory/` — 선호/규칙/프로젝트 핵심 정보 양방향 동기화 |

> **lite 모드**: Obsidian Vault 없이 로컬 파일시스템(`~/.doma/`)만으로 동작. 교차 연결 기능 제한.

사용자가 뭔가를 기억/저장/기록해달라고 하면, 내용의 성격에 따라 올바른 저장소로 자동 라우팅한다.

## Vault 경로

Primary: `{OBSIDIAN_VAULT}/`

Desktop Commander의 `read_file`, `edit_block`, `write_file`, `create_directory` 도구를 사용.

## 1차 라우팅: 어디에 저장할 것인가

| 유형 | 판별 | 저장 위치 |
|------|------|----------|
| **일정/약속** | 날짜+시간, "미팅", "약속", "일정" | **Google Calendar** (`gcal_create_event`) |
| **프로젝트 관련** | 프로젝트명/업무 내용 언급 | Obsidian → `L3-Active/Work/{프로젝트명}/` (아래 2차 라우팅) |
| **업무 태스크** | "해야 해", "보내야 해" (프로젝트 특정 안 됨) | Obsidian → `Action Tracker.md` |
| **의사결정** | "~로 결정", "~안으로", "확정" | Obsidian → `Decision Log.md` + 프로젝트 README 결론 섹션 |
| **외주사/파트너** | 회사명 + 연락처/담당자/계약 | Obsidian → `L1-Knowledge/People/` |
| **브랜드/톤앤매너** | "브랜드", "톤", "보이스", "스타일" | Obsidian → `L2-Methods/Work/` |
| **아이디어/메모** | 위 어디에도 해당 안 됨 | Obsidian → `L4-Buffer/Inbox/` |
| **자동화 요청** | "자동화해줘", "스킬 만들어" | **skill-creator 스킬** 호출 |

## 2차 라우팅: 프로젝트 내 어디에 저장할 것인가

프로젝트 관련 내용이면, 대화 내용의 성격을 분석해서 README.md의 해당 섹션 또는 하위 파일에 저장한다.

### 프로젝트 폴더 구조

```
L3-Active/Work/{프로젝트명}/
├── README.md          ← 마스터 문서 (모든 메타정보)
├── 회의록/            ← YYMMDD-{주제}.md
├── 자료조사/          ← 리서치, 벤치마킹
├── 레퍼런스/          ← 참고 사례, 링크
└── 아웃풋/            ← 산출물 기록
```

### README.md 섹션별 분류

사용자의 대화 내용을 분석해서, 아래 섹션 중 해당하는 곳에 append:

| 사용자가 말하는 내용 | README 섹션 | 형식 |
|---------------------|------------|------|
| 주체, 담당자, 예산, 기간, 데드라인 | **기본 정보** | 메타 필드 업데이트 |
| 목적, 배경, 범위, 업무 설명 | **프로젝트 내용** | 서술 append |
| 디자인 방향, 전략, 톤앤매너, 크리에이티브 | **컨셉** | 서술 append |
| 브레인스토밍, 시도할 것, 떠오른 생각 | **아이디어** | bullet 추가 |
| "~차 리뷰", "진행 중", 상태 업데이트 | **진행상황** | 테이블 row 추가 |
| 시안, 결과물, 가이드라인, 버전 | **아웃풋 프로세스** | 테이블 row 추가 |
| "~로 결정", "~안 확정", 합의 내용 | **결론/결정사항** | bullet 추가 |
| "~해야 해", 할 일, 다음 액션 | **TODO 리스트** | 체크박스 추가 |
| 확인/검수/QC/체크 항목 | **체크리스트** | 체크박스 추가 |
| 연관 프로젝트 언급 | **관련 프로젝트** | 링크 추가 |

### 별도 파일이 필요한 경우

내용이 길거나 독립적인 경우, README 대신 하위 폴더에 별도 파일을 만든다:

| 내용 | 저장 위치 | 파일명 규칙 |
|------|----------|------------|
| 미팅/회의 상세 내용 | `회의록/` | `YYMMDD-{주제}.md` |
| 리서치, 벤치마킹, 데이터 | `자료조사/` | `{주제}.md` |
| 참고 사례, 이미지, URL | `레퍼런스/` | `{주제}.md` |
| 산출물 상세 (시안 피드백 등) | `아웃풋/` | `{산출물명}.md` |

판단 기준: 3줄 이하 → README 해당 섹션에 직접. 3줄 초과 또는 독립 문서 성격 → 하위 파일.

## 기존 단일 파일 마이그레이션

`L3-Active/Work/`에 이미 단일 .md로 존재하는 프로젝트는, 사용자가 해당 프로젝트에 새 정보를 추가할 때 점진적으로 폴더 구조로 전환. 기존 내용은 README.md의 적절한 섹션으로 재배치.

---

## 실행 순서

1. **1차 분류**: 일정 / 프로젝트 / 태스크 / 결정 / 기타
2. **2차 분류** (프로젝트인 경우): README 섹션 or 하위 파일
3. **기존 파일 확인**: 중복 방지, 해당 프로젝트 폴더 존재 여부
4. **저장**: 적절한 위치에 기록
5. **태깅**: 자동 태그 부여
6. **갱신**: README.md `updated` 날짜, `status` 반영
7. **보고**: 한 줄 알림

```
✅ 프로젝트 → L3-Active/Work/스톤-미팅/README.md [기본 정보] 업데이트 완료 #스톤
✅ 프로젝트 → L3-Active/Work/클레이-브랜딩/회의록/260326-리뷰.md 생성 완료 #클레이 #BX
✅ 일정 → Google Calendar "3/28 클레이 미팅 14:00" 등록 완료
✅ 태스크 → Action Tracker [긴급] "시안 피드백 전달" 추가 완료
```

## 주체 불분명 시 판별 로직

프로젝트 주체를 모를 때, 순서대로 시도한다:

**1순위 — 대화 맥락 추론**
직전 대화에서 특정 프로젝트/업체를 다루고 있었으면 그걸 주체로 판단.
예: 클레이 얘기하다가 "이거 기억해" → 주체 = 클레이

**2순위 — 키워드 매칭 (기존 프로젝트 히스토리)**
| 키워드 | 추론 주체 |
|--------|----------|
| 브랜딩, 표지, 가이드라인, BI | 클레이 |
| 스케줄러, VOD, 온라인, 플랫폼 | {회사명} (닷컴) |
| 교재, 출판, 북스 | {회사명} (BOOKS) |
| 논술, {프로젝트명} | {프로젝트명} |
| UX, UI, 웹, 앱 | UX팀 |
| 사이니지, 기숙학원, 용인 | {회사명} (공간) |
| 굿즈, 웰컴키트, 트롤리 | 기획디자인팀 |
| 서체, 폰트 | {회사명} (전용서체) |

**3순위 — 기본값: 기획디자인팀**
{USER}의 본업 팀이 기획디자인팀이므로, 위 두 단계로 판별 불가 시 기본값으로 설정.
README에 `⚠️ 주체 미확인 — 추후 수정 필요` 표시를 남긴다.

**최후수단 — 짧게 한 번만 질문**
완전히 새로운 프로젝트이고, 맥락도 키워드도 없을 때만:
"주체가 어디야? ({회사명}/클레이/기획디자인팀/기타)"

이 판별 로직은 프로젝트 주체뿐 아니라, 브랜드(#BOOKS #닷컴 #LIVE) 태그 결정에도 동일하게 적용한다.

## 중요 원칙

- **질문 최소화**: 분류가 명확하면 바로 실행. 주체 판별도 최대한 자동.
- **중복 체크**: 같은 내용이 이미 있으면 추가하지 않고 알림
- **마감일 추론**: "이번 주까지" → 구체적 날짜로 변환
- **점진적 구조화**: 기존 파일은 건드리지 않고, 새 정보 추가 시에만 전환
- **양쪽 기록**: 결정은 Decision Log + 프로젝트 README 둘 다. TODO는 Action Tracker + 프로젝트 README 둘 다.
- **미확인 표시**: 추론으로 넣은 정보에는 ⚠️ 표시해서 나중에 사용자가 확인/수정 가능하게.

## Claude Memory 동기화 (8단계 — 모든 저장 후 자동 실행)

기억해/메모해/저장해 요청은 Obsidian 저장 뿐 아니라, Claude의 자체 memory에도 반영해야 집컴/회사컴 어디서든 Claude가 기억한다.

### 판별: Claude memory에도 저장해야 하는가?

| 유형 | Claude memory 저장 | 이유 |
|------|-------------------|------|
| 사용자 선호/스타일 | O (user 타입) | 모든 대화에 영향 |
| 행동 규칙/피드백 | O (feedback 타입) | 같은 실수 반복 방지 |
| 프로젝트 핵심 정보 | O (project 타입) | 프로젝트 맥락 유지 |
| 외부 시스템 위치 | O (reference 타입) | 정보 찾기 경로 |
| 일회성 태스크 | X | Action Tracker만으로 충분 |
| 일정/약속 | X | Google Calendar만으로 충분 |
| 상세 회의록 | X | Obsidian만으로 충분 |

### 실행 순서

```
1. Obsidian에 저장 (기존 1~7단계)
2. Claude memory에도 저장이 필요한 유형인지 판별
3. 필요하면:
   a. ~/.claude/projects/-Users-{USER}/memory/ 에 파일 생성/업데이트
   b. MEMORY.md 인덱스에 한 줄 추가
   c. 이 경로는 심볼릭 링크로 {SYNC_DIR}에 연결되어 있으므로
      → Google Drive 자동 동기화 → 집컴에서도 즉시 사용 가능
4. 보고에 동기화 상태 포함
```

### 보고 형식 (기존 형식에 추가)

```
[Obsidian] L3-Active/Work/클레이-브랜딩/README.md [컨셉] 업데이트 완료
[Memory] project_clay_branding.md 동기화 완료 (집컴 자동 반영)
```

### memory 파일 작성 규칙

기존 auto memory 시스템의 프론트매터 형식을 따른다:

```markdown
---
name: {메모리 이름}
description: {한 줄 설명 — 미래 대화에서 관련성 판단용}
type: {user|feedback|project|reference}
---

{내용}
```

### 기존 memory 업데이트 우선

새 파일을 만들기 전에 MEMORY.md를 읽고, 같은 주제의 기존 memory가 있으면 새로 만들지 말고 업데이트한다.

### 동기화 경로 (참고)

```
~/.claude/projects/-Users-{USER}/memory/  (심볼릭 링크)
  → Google Drive/{SYNC_DIR}/memory/     (실제 저장소)
  → 집컴 Google Drive 동기화            (자동)
  → 집컴 ~/.claude/.../memory/          (심볼릭 링크)
```

수정 즉시 양방향 동기화. 별도 sync 명령 불필요.
