# Setup Team v2 (프로젝트 팀 자동 구성기)

프로젝트 유형을 분석해서 필요한 에이전트들을 자동으로 배치하고, 각각의 역할과 작업 분배 규칙을 지정하는 스킬이다.

전체 에이전트 목록과 역할은 `{SYNC_DIR}/agents/README.md`에서 확인한다.

---

## 프리셋 팀 구성 — 유형별 즉시 배치

### 프리셋 1: 디자인 프로젝트

```
적용 조건: 브랜드 디자인, 슬라이드, 카드뉴스, 배너, PPT, UI/UX 작업

PM
└── design-master-director (디자인 총괄)
    ├── ui-designer (UI 컴포넌트, 레이아웃)
    ├── visual-designer (컬러, 타이포, 그래픽)
    └── motion-designer (애니메이션, 전환 효과)

검토: red-team (방향성 검증 — "이 디자인이 목적에 맞는가?")
선택: mobile-ux (모바일 화면 포함 시 추가)

PM의 작업 분배 규칙:
  1. design-master-director에게 전체 디자인 방향 위임
  2. 각 전문가에게 하위 작업 배정 (UI/시각/모션 분리)
  3. red-team이 "고객이 이해하는가?" 관점에서 검토
  4. PM이 최종 방향 승인 후 수정 지시
```

### 프리셋 2: 사업 검토 / 아이디어 검증

```
적용 조건: 사업 아이디어, 수익 모델, 시장 진입 전략, GO/PIVOT/STOP 판정

PM (Trinity Check 자동 발동)
├── [Trinity Check 1단계]
│   ├── mvp-architect ("이거 팔 수 있어?")
│   └── red-team ("이게 진짜야?")
├── [페르소나 시뮬레이션 2단계]
│   ├── persona-business (사업가 관점)
│   └── persona-critic (까다로운 리뷰어 관점)
└── [전략 실행 3단계]
    ├── strategy-lead → market-analyst + competitor-analyst
    └── killer-finder (킬러 서비스 1가지 선정)

PM의 작업 분배 규칙:
  1. Trinity Check 먼저. 사용자 개입 없이 내부 처리.
  2. GO 판정 시에만 strategy-lead 팀 투입
  3. STOP이면 이유와 함께 즉시 전달. 더 진행 안 함.
  4. PIVOT이면 수정 방향과 함께 재검토
```

### 프리셋 3: 코드 프로젝트 (웹앱 / 풀스택)

```
적용 조건: 기능 구현, 새 페이지 개발, API 연동, 풀스택 웹앱

PM
└── code-builder (개발 리드)
    ├── frontend-dev (화면, 컴포넌트, 상태)
    ├── backend-dev (API, DB, 외부 연동)
    └── test-engineer (QA, 테스트 코드)

검토: code-reviewer + security-lead (배포 전 필수)
배포: deployer + infra-engineer
선택: mobile-dev + mobile-perf (모바일 중요 프로젝트)

PM의 작업 분배 규칙:
  1. code-builder가 아키텍처(전체 구조) 결정 → PM 승인
  2. 기능 단위로 쪼개서 frontend/backend 병렬 배정
  3. test-engineer는 기능 완성 즉시 검증 (배포 전 대기 금지)
  4. code-reviewer + security-lead 통과 후에만 deployer 호출
  5. quality-director Go/No-Go → PM 최종 승인 → 배포
```

### 프리셋 4: 출시 전 최종 검수

```
적용 조건: 배포 직전, 품질 게이트

quality-director (총괄)
├── code-reviewer (코드 품질)
├── security-lead (보안)
├── mobile-tester (모바일 실기기)
└── red-team (최후 구멍 찾기)

판정: Go → deployer 호출 / No-Go → 이슈 목록 + 수정 지시
```

### 프리셋 5: 리서치 + 경쟁사 분석

```
적용 조건: 시장 조사, 경쟁사 분석, 기술 스택 결정

research-lead (총괄)
├── market-analyst (시장 규모, 타겟)
├── competitor-analyst (유사 서비스)
└── tech-researcher (기술 선택)

경쟁 인텔리전스 추가 시:
competitive-intel-lead
├── competitor-teardown (기능/플로우/스택 해부)
├── ux-benchmarker (디자인 비교)
└── strategy-counter (약점 공략)
```

### 프리셋 6: 수익화 설계

```
적용 조건: 가격 설정, 결제 연동, 수익 모델

monetization-lead
├── pricing-strategist (가격 구조 설계)
├── revenue-analyst (수익 지표)

```

## 실행 프로세스

### Step 1: 프로젝트 분석

현재 프로젝트 폴더를 분석한다:
- `CLAUDE.md` 읽기 → 프로젝트 설명 파악
- `package.json` 확인 → 기술 스택 파악
- `requirements.txt` 확인 → Python 여부
- 폴더 구조 확인 → 프로젝트 규모 판단
- 만약 프로젝트 폴더가 없으면, 사용자에게 뭘 만들 건지 물어본다

### Step 2: 프리셋 매칭 또는 수동 구성

위 6개 프리셋 중 가장 가까운 것을 선택한다. 복합 프로젝트면 여러 프리셋을 조합한다.

프리셋 매핑:
| 요청 키워드 | 프리셋 |
|-----------|-------|
| 디자인, 브랜드, 슬라이드, UI, PPT, 배너, 카드뉴스 | 프리셋 1 |
| 사업 아이디어, 검증, 수익 모델, 성공할까, GO/STOP | 프리셋 2 |
| 앱 만들어, 기능 추가, 코드, 웹사이트 개발 | 프리셋 3 |
| 배포하기 전에, 출시 전 점검, 품질 확인 | 프리셋 4 |
| 시장 조사, 경쟁사, 어떤 기술 쓸지 | 프리셋 5 |
| 얼마로 팔지, 결제 연동, 수익화 | 프리셋 6 |

### Step 3: PM 작업 분배 계획 수립

PM이 팀 배치 후 다음을 자동으로 정의한다:

```
작업 분배 계획:
  [작업 1] {에이전트명} → {구체적 산출물} → 완료 기준: {기준}
  [작업 2] {에이전트명} → {구체적 산출물} → 완료 기준: {기준}
  [검토] {검토 에이전트} → 통과 조건: {조건}
  [완료] PM 최종 확인 → 사용자 전달

병렬 가능: [작업 1] + [작업 2] 동시 진행
순차 필수: [검토]는 [작업 1,2] 완료 후
```

### Step 4: 진행 상황 추적 규칙

PM은 다음 형식으로 진행 상황을 관리한다:

```
스프린트 상태:
  진행 중: {에이전트명} — {현재 작업}
  대기 중: {에이전트명} — {선행 작업 완료 대기}
  완료:   {에이전트명} — {산출물} [완료]
  블록:   {에이전트명} — {블로킹 이슈} [해결 필요]
```

블로킹 이슈 발생 시:
1. PM이 즉시 인지하고 원인 파악
2. 2가지 해결 옵션 제시 (우회 or 직접 해결)
3. 10분 내 결정. 결정 안 되면 PM이 정함

### Step 5: 결과물 취합 + 품질 검수 체크리스트

팀 작업 완료 후 PM이 아래 체크리스트를 실행한다:

```
[결과물 완성도]
  [ ] 요청한 기능/산출물이 모두 포함됐는가?
  [ ] 빈 상태(데이터 없을 때), 로딩, 에러 — 모두 처리됐는가?
  [ ] 모바일(375px)에서 문제 없는가? (모바일 프로젝트 필수)

[코드 품질] (코드 프로젝트 해당)
  [ ] code-reviewer 통과됐는가?
  [ ] security-lead 보안 점검 완료됐는가?
  [ ] 브라우저 콘솔에 에러가 없는가?

[디자인 품질] (디자인 프로젝트 해당)
  [ ] Apple HIG 5원칙 (명확성, 피드백, 콘텐츠 우선, 일관성, 심미성) 충족하는가?
  [ ] 브랜드 톤과 맞는가?
  [ ] 초보자가 3초 안에 이해하는가?

[사업/전략 품질] (사업 검토 해당)
  [ ] Trinity Check 3단계 모두 완료됐는가?
  [ ] 실행 항목이 구체적인가? (누가 / 언제 / 뭘)
  [ ] STOP 사유가 있으면 대안이 제시됐는가?

[최종 판정]
  모두 통과 → quality-director Go 사인 → PM 사용자 전달
  미통과 항목 있음 → 해당 에이전트에 수정 지시 → 재검수
```

### Step 6: 프로젝트 CLAUDE.md에 팀 정보 추가

프로젝트의 CLAUDE.md에 아래 섹션을 추가한다:

```markdown
## 팀 구성

이 프로젝트에 배치된 에이전트 팀:

| 역할 | 에이전트 | 담당 |
|------|---------|------|
| {역할} | {에이전트명} | {구체적 담당 업무} |

### 사용법
- "디자인 바꿔줘" → design-master-director가 처리
- "기능 추가해줘" → code-builder가 처리
- "에러 나" → bug-fixer가 처리
- "인터넷에 올려줘" → deployer가 처리
- "코드 괜찮은지 봐줘" → code-reviewer가 처리
- "이게 뭐야?" → beginner-helper가 설명
- "사업성 검토해줘" → pm이 Trinity Check 발동

### 다음 할 일
{PM이 작성하는 구체적 다음 액션}
```

### Step 7: 완료 보고

아래 형식으로 결과를 알려준다:

```
팀 구성 완료.

프로젝트: {프로젝트명}
유형: {프리셋명}

배치된 팀:

  {에이전트명} ({직책})
     담당: {구체적 업무}
     결과물: {산출물}

작업 순서:
  1단계 (병렬): {에이전트A} + {에이전트B}
  2단계 (순차): {검토 에이전트}
  3단계: PM 최종 확인

시작하려면 "시작해줘" 또는 첫 번째 작업을 직접 말하면 된다.
```

## 중요 원칙

1. **자동 판단**: 프로젝트 파일을 보고 알아서 팀을 구성한다. 사용자에게 "어떤 에이전트를 쓸까요?" 묻지 않는다.
2. **PM이 분배 계획을 명시적으로 작성한다**: "알아서 해줘"가 아니라 누가 뭘 언제 하는지 구체적으로 적는다.
3. **검토는 항상 분리**: 만드는 사람과 검토하는 사람이 다르다. code-builder가 만들면 code-reviewer가 검토.
4. **최소 팀 원칙**: 필요한 에이전트만 투입. 많을수록 산만해진다. 2~5명이 적정.
5. **beginner-helper는 코드 프로젝트에 항상 포함**: 초보자이므로 모든 개발 팀에 통역사 역할 배치.
6. **Trinity Check 자동 발동**: 사업성 판단 / 가격 설정 / 킬러 선정 요청이 오면 PM이 자동으로 발동시킨다.
