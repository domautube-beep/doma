# Design Pipeline v1.0

디자인 요청이 들어오면 **단계별로 관리**한다. 시안 하나로 끝나지 않고, 접수 → 확정 → 아카이브까지 끊김 없이.

> design-system-builder = 한 장면을 만드는 법
> design-pipeline = 프로젝트 전체를 운영하는 법

---

## 7단계 파이프라인

### 1단계. 접수 (Intake)

사용자가 디자인 요청을 말하면:

1. 요청 내용을 아래 구조로 정리한다:
   ```
   - 프로젝트명:
   - 매체/형식: (웹/앱/배너/카드뉴스/브랜드/패키지...)
   - 타깃:
   - 핵심 메시지:
   - 참고 자료: (없으면 없음)
   - 마감:
   - 수정 횟수 포함 여부:
   ```

2. Experience Intent를 정의한다:
   ```
   - 이 디자인으로 사람들이 무엇을 하게 되는가
   - 3초 안에 전달해야 할 것 한 가지
   - 느껴야 할 감정 / 느끼면 안 되는 감정
   ```

3. Obsidian에 프로젝트 노트를 생성한다:
   - 경로: `L3-Active/Work/{프로젝트명}.md`
   - 포함 내용: 요청 정리 + Experience Intent + 진행 상태

4. 파일 구조를 만든다:
   ```
   Google Drive/01 작업실/{프로젝트명}/
   ├── design-brief.md      ← 요청 + Experience Intent
   ├── feedback-log.md      ← 피드백 기록 (빈 파일로 시작)
   └── outputs/             ← 산출물 폴더
   ```

5. 바로 2단계로 진입한다. "확인됐습니다" 같은 중간 멘트 금지.

---

### 2단계. 시안 생성 (Draft)

design-system-builder 스킬의 7단계 마스터 프로세스를 실행한다:

1. Experience Intent 기반으로 시안 방향 결정
2. `design-system-builder` 스킬로 HTML 시안 생성
3. 파일 저장:
   - `/tmp/{프로젝트명}-v1.html` 또는
   - Google Drive 경로 `01 작업실/{프로젝트명}/outputs/v1.html`
4. 브라우저로 바로 열기: `open {파일경로}`
5. 시안과 함께 아래를 같이 전달한다:
   ```
   [시안 v1]
   - 적용한 Experience Intent: ...
   - 주요 디자인 결정 3가지: ...
   - 수정 요청 방법: "피드백: {내용}" 형식으로
   ```

---

### 3단계. 피드백 수집 (Feedback)

사용자가 "피드백:" 또는 "수정:" 또는 구체적인 수정 요청을 하면:

1. 피드백을 아래 구조로 분류한다:
   ```
   [좋은 것] 유지할 것
   [나쁜 것] 제거할 것
   [수정할 것] 바꿀 것 + 방향
   [추가할 것] 새로 넣을 것
   ```

2. `feedback-log.md`에 기록한다:
   ```markdown
   ## v{N} → v{N+1} 피드백 ({날짜})
   ### 좋은 것
   - ...
   ### 수정 사항
   - ...
   ### 결정: {왜 이렇게 수정하는가}
   ```

3. 바로 4단계(수정)로 진입한다.

---

### 4단계. 수정 (Revision)

1. 피드백을 반영해 시안을 수정한다
2. 버전 번호를 올린다: v1 → v2 → v3...
3. 저장: `outputs/v{N}.html`
4. 전달 시 변경 내용 명시:
   ```
   [시안 v{N}] — v{N-1} 대비 변경사항
   - 여백 40px → 80px 확장
   - 헤드라인 폰트 Playfair Display로 변경
   - 배경 그라디언트 방향 수정
   ```
5. 피드백이 계속 오면 3단계 → 4단계를 반복한다
6. 버전이 3회 이상이면 "확정 또는 추가 수정" 선택지를 제시한다

---

### 5단계. 확정 (Confirm)

사용자가 "확정", "OK", "이걸로" 같은 말을 하면:

1. 최종 파일 저장:
   - `outputs/final.html` (복사본)
2. PDF 변환:
   ```bash
   npx playwright pdf {파일경로} ~/Downloads/{프로젝트명}-final.pdf
   ```
   playwright가 없으면:
   ```bash
   /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
     --headless --print-to-pdf=~/Downloads/{프로젝트명}-final.pdf \
     {파일경로}
   ```
3. Google Drive에도 저장:
   - `01 작업실/{프로젝트명}/outputs/final.pdf`
4. 확정 내용을 Obsidian에 기록:
   - `L3-Active/Work/{프로젝트명}.md`에 "확정: {날짜}, 최종버전: v{N}" 추가
5. 바로 6단계(핸드오프) 여부를 묻는다:
   ```
   최종본 확정됐습니다.
   핸드오프 문서가 필요하면 "핸드오프" 라고 하세요.
   바로 아카이브할게요.
   ```

---

### 6단계. 핸드오프 (Handoff)

사용자가 "핸드오프"를 요청하면:

1. `design-handoff` 스킬을 실행한다
2. 대상에 따라 문서 유형 선택:
   - 클라이언트 전달용 → 시안 설명서
   - 개발자 전달용 → 개발 핸드오프
   - 둘 다 → 두 문서 모두
3. 생성된 문서를:
   - `outputs/handoff.md` 저장
   - Obsidian `L3-Active/Work/{프로젝트명}.md`에 첨부
4. Figma 파일이 있으면 MCP로 디자인 정보 추출해서 문서에 반영

---

### 7단계. 아카이브 (Archive)

핸드오프 완료 또는 사용자가 "아카이브" 요청 시:

1. Obsidian 프로젝트 노트에 완료 태그 추가:
   ```
   상태: 완료
   완료일: {날짜}
   최종 파일: {경로}
   ```
2. Google Drive 폴더 이름 앞에 "완료-" 추가:
   `01 작업실/완료-{프로젝트명}/`
3. Decision Log에 기록:
   ```
   프로젝트: {프로젝트명}
   주요 결정: {핵심 디자인 결정 1-2개}
   결과: {클라이언트 반응 또는 활용 현황}
   ```
4. 완료 선언:
   ```
   [{프로젝트명}] 아카이브 완료.
   총 {N}회 수정, {N}일 소요.
   ```

## 상태 전환 트리거

| 사용자 발언 | 진입 단계 |
|------------|----------|
| "{프로젝트명} 디자인 시작" / "{프로젝트명} 만들어줘" | 1단계 |
| "피드백:", "수정:", "바꿔줘" | 3단계 → 4단계 |
| "확정", "OK", "이걸로", "최종" | 5단계 |
| "핸드오프", "개발자한테 전달", "클라이언트 전달" | 6단계 |
| "아카이브", "완료", "끝났어" | 7단계 |
| "현황", "진행상황", "어디까지 왔어" | 현재 상태 요약 출력 |

## 파일 구조

```
Google Drive/01 작업실/{프로젝트명}/
├── design-brief.md          ← 요청 + Experience Intent
├── feedback-log.md          ← 피드백 이력
└── outputs/
    ├── v1.html
    ├── v2.html
    ├── v3.html (있으면)
    ├── final.html
    ├── final.pdf
    └── handoff.md

Obsidian/L3-Active/Work/{프로젝트명}.md
    ← 접수 → 진행 → 확정 → 아카이브 전체 이력
```

---

## 버전 관리 원칙

- v1, v2, v3... 순서로 덮어쓰지 않고 누적 저장
- 각 버전은 왜 바뀌었는지 feedback-log.md에 기록
- 이전 버전으로 롤백 요청 시: 해당 v{N}.html 파일 재오픈

---

## design-system-builder와의 관계

```
design-pipeline  ←  프로젝트 관리 레이어 (언제, 뭘, 어떻게 저장할지)
      ↓
design-system-builder  ←  실제 디자인 제작 레이어 (어떻게 만들지)
      ↓
design-handoff   ←  문서화 레이어 (어떻게 전달할지)
```

design-pipeline은 다른 두 스킬을 **호출**한다. 직접 만들지 않는다.

---

## 자기 검수

```
□ 프로젝트 노트가 Obsidian에 있는가
□ feedback-log.md에 모든 피드백이 기록됐는가
□ 버전 파일이 누적 저장됐는가 (덮어쓰기 금지)
□ 확정 시 Downloads + Drive 두 곳에 PDF가 있는가
□ 아카이브 시 Decision Log에 기록했는가
□ 다음 단계가 항상 명확하게 안내됐는가
```
