# Finish (작업 마무리)

구현이 끝난 후 깔끔하게 마무리하는 스킬이다.
변경사항 요약 → 커밋 → PR 또는 병합까지 자동 제안한다.

## 언제 사용하나

- 기능 구현이 끝났을 때
- 사용자가 "다 됐어", "마무리해줘", "커밋해줘" 라고 할 때
- `/finish` 명령어를 직접 호출할 때

## 실행 순서

### Step 1: 변경사항 분석

git 상태를 확인하고 변경 내역을 정리한다.

```bash
git status
git diff --stat
git diff
git log --oneline -5
```

### Step 2: 변경 요약 보고

아래 형식으로 사용자에게 보고한다:

```
## 작업 완료 요약

### 변경된 파일 ({N}개)
- `{파일1}` — {뭘 했는지 한 줄}
- `{파일2}` — {뭘 했는지 한 줄}

### 추가된 파일 ({N}개)
- `{파일1}` — {역할}

### 삭제된 파일 ({N}개)
- `{파일1}` — {삭제 이유}

### 핵심 변경
{이번 작업에서 가장 중요한 변경을 2-3줄로 설명}
```

### Step 3: 마무리 옵션 제안

프로젝트 상태에 따라 적절한 옵션을 제안한다:

#### Git 저장소인 경우

```
어떻게 마무리할까요?

1. **커밋만** — 로컬에 저장 (추천: 아직 작업 중일 때)
2. **커밋 + PR 생성** — GitHub에 Pull Request 생성 (추천: 리뷰 받을 때)
3. **커밋 + main 병합** — 바로 main에 합치기 (추천: 혼자 작업할 때)
4. **나중에** — 지금은 아무것도 안 함
```

#### Git 저장소가 아닌 경우

```
어떻게 마무리할까요?

1. **Git 초기화 + 첫 커밋** — 버전 관리 시작
2. **나중에** — 지금은 아무것도 안 함
```

### Step 4: 선택에 따라 실행

#### 옵션 1: 커밋만
```bash
git add {변경된 파일들}
git commit -m "{자동 생성된 커밋 메시지}"
```

커밋 메시지 규칙:
- 첫 줄: 한국어로 핵심 변경 요약 (50자 이내)
- 빈 줄
- 본문: 변경 파일과 내용 목록
- 마지막 줄: `Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>`

#### 옵션 2: 커밋 + PR
```bash
git checkout -b {feature/기능명}
git add {변경된 파일들}
git commit -m "{커밋 메시지}"
git push -u origin {브랜치명}
gh pr create --title "{PR 제목}" --body "{자동 생성된 PR 본문}"
```

PR 본문에 포함할 것:
- `## Summary` — 변경 요약 (2-3줄)
- `## Changes` — 파일별 변경 내역
- `## Test plan` — 확인 방법

#### 옵션 3: 커밋 + main 병합
```bash
git add {변경된 파일들}
git commit -m "{커밋 메시지}"
```
(이미 main에 있으면 커밋만. 브랜치에 있으면 main으로 병합)

### Step 5: 완료 보고

```
마무리 완료!

- 커밋: {커밋 해시 앞 7자리}
- 브랜치: {브랜치명}
- PR: {PR URL} (PR 생성한 경우)

다음에 할 수 있는 것:
- "새 기능 추가해줘"
- "버그 있어" → 수정
- "/brainstorm" → 다음 작업 계획
```

## 중요 원칙

1. **민감 파일 제외**: `.env`, 비밀키, 인증 파일은 절대 커밋하지 않는다. 발견 시 경고.
2. **구체적 파일 add**: `git add -A` 대신 파일을 하나씩 지정한다.
3. **사용자 확인 필수**: 커밋/PR/병합 전에 반드시 사용자 동의를 받는다.
4. **force push 금지**: `--force` 옵션은 사용하지 않는다.
5. **브랜치 보호**: main/master에 직접 force push하지 않는다.
