# Brainstorm (구현 전 브레인스토밍)

코딩을 시작하기 **전에** 반드시 거쳐야 하는 사고 단계다.
Superpowers 방식: 생각 → 계획 → 확인 → 실행.

## 언제 사용하나

- 사용자가 새 기능, 새 프로젝트, 큰 변경을 요청할 때
- "어떻게 만들지", "설계해줘", "계획 세워줘" 라고 할 때
- `/brainstorm` 명령어를 직접 호출할 때

## 실행 순서

### Phase 1: 문제 이해 (30초)

사용자의 요청을 분석해서 아래 3가지를 정리한다:

```
## 문제 정의
- **뭘 만드나**: {한 줄 요약}
- **누가 쓰나**: {사용자/대상}
- **성공 기준**: {이걸 만들면 뭐가 달라지나}
```

### Phase 2: 접근법 비교 (1분)

최소 2개, 최대 3개의 접근법을 제시한다.
각 접근법마다 **장점, 단점, 예상 시간**을 표로 비교한다.

```
## 접근법 비교

| | 방법 A: {이름} | 방법 B: {이름} | 방법 C: {이름} |
|---|---|---|---|
| 핵심 | {한 줄} | {한 줄} | {한 줄} |
| 장점 | {구체적} | {구체적} | {구체적} |
| 단점 | {구체적} | {구체적} | {구체적} |
| 난이도 | 쉬움/보통/어려움 | 쉬움/보통/어려움 | 쉬움/보통/어려움 |

**추천: 방법 {X}** — {이유}
```

### Phase 3: 실행 계획 (1분)

추천 방법을 기준으로 구체적 실행 계획을 작성한다.

```
## 실행 계획

### 만들 파일
1. `{파일경로}` — {역할}
2. `{파일경로}` — {역할}

### 작업 순서
1. {첫 번째 할 일} (예상: N분)
2. {두 번째 할 일} (예상: N분)
3. {세 번째 할 일} (예상: N분)

### 필요한 것
- {라이브러리, API 키, 외부 서비스 등}

### 위험 요소
- {실패할 수 있는 지점과 대안}
```

### Phase 4: 사용자 확인

반드시 사용자에게 확인을 받는다:

```
이 계획대로 진행할까요?
- "ㅇㅇ" → 바로 구현 시작
- "B로 해" → 방법 B로 계획 수정
- "{수정사항}" → 계획 수정 후 재확인
```

## 중요 원칙

1. **코딩 금지**: 이 스킬 실행 중에는 코드를 작성하지 않는다. 계획만 세운다.
2. **구체적 숫자**: "빠르다"가 아니라 "약 5분", "약 3개 파일" 처럼 구체적으로.
3. **현실적 평가**: 장점만 나열하지 않는다. 단점과 위험도 솔직하게.
4. **기존 코드 확인**: 프로젝트가 있으면 기존 구조를 먼저 파악한 후 계획한다.
5. **1회 질문**: 계획에 필요한 추가 정보가 있으면 Phase 1에서 한 번만 묻는다.
